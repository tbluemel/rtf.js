1 INTRODUCTION
Process Name Technology Solution System Integration Testing (SIT) Process Process Owner Manager, Business Solution Services
2 PURPOSE
The Technology Solution System Integration Testing (SIT) process validates that the technology solution and its features conform to the Technology Solution Design document specifications and the Technology Solution Requirements, prior to the customer testing.
The objectives of the Technology Solution System Integration Testing Process are:
1. To validate that the technology solution meets all documented requirements.
2. To identify and correct problems in the technology solution before it is installed in the Customer Acceptance Testing (CAT) environment.
3. To determine how the technology solution measures against technology service provider expectations.
4. To ensure no increase in risk to existing Postal Service infrastructure and applications.
3 SCOPE
The SIT process includes, but is not limited to, the following :
1. All technology solutions and their components ( hardware, software, database, and/or network) to be authorized in the approved Technology Solution Requirements document.
2. All services (including network, server, and mainframe) to be deployed in the Postal Service Technical Environment.
3. All database management systems (DBMS) to be developed to operate in the Postal Service Technical Environment.
4. All maintenance releases for services and technology solutions to be operable in the Postal Service Technical Environment. Maintenance releases include hardware, software, network, and DBMS upgrades.
4 PROCESS DESCRIPTION
The Technology Solution System Integration Testing Process consists of the following sub-processes.
1. Create Draft SIT Strategy
The Draft SIT Strategy is developed early and is input to the Technology Solution Analysis & Design process to support generation of accurate and committed cost and schedule. The Draft SIT Strategy will include the following information:
a. SIT logistics (where)
b. People resource requirements (customer, IT, etc.)
c. Data requirements (source & migration)
d. High level script strategy (functionality, performance, etc.)
e. Hardware requirements strategy
f. Timeframe
g. Backup requirements
h. Security requirements
2. Create SIT Readiness Plan
This is the detailed plan that supports the SIT Strategy that was developed during the Design and Analysis phase. Any change to the SIT Strategy at this time must be approved by the CCB. The SIT Readiness Plan is a project plan that addresses the following activities that are needed to perform SIT:
a. Data setup
b. Problem Log setup
c. SIT room setup
d. Code promotion
e. Development of job streams for backup and test execution
f. People readiness
3. Create Final SIT Strategy
To create the Final SIT Strategy, the Draft SIT Strategy will be revisited and updated where appropriate with the most current information available. The Final SIT Strategy will include current information from the Technology Solution Requirements, Analysis & Design, and Build processes.
4. Finalize the SIT Scripts
Finalize the SIT Scripts to ensure that requirements are thoroughly tested.
5. Create SIT Plan
The SIT Plan will include all steps necessary to perform SIT. This includes when and who will exercise which scripts, the system jobs to be run, system backups, security procedures to be followed, etc.
6. Perform SIT
Execute the SIT Plan. All SIT test results are documented on the scripted materials, identified as problems on the problem log, or identified as new requirements on the enhancement log. All test results should be retained in accordance with Postal Service data retention requirements.
7. Analyze SIT Test Results
Review the SIT test results and determine if the technology solution meets the Technology Solution Requirements. Problems identified in testing will be categorized and documented in either a problem or an enhancement log. If the issue results in a change to the accepted Technology Solution Requirements or Design documents, it will be logged and submitted to the CCB. Problems will be entered, prioritized, and resolved prior to the CAT process.
8. SIT Documented Approval
The purpose of this activity is to create a formal checkpoint and documentation of acceptance of the as-built tested technology solution. Upon acceptance, an acceptance letter is issued and signed by all Service Providers. The accepted technology solution may now begin its migration to the CAT environment through the release management process.
9. Conduct Security Code Review Report
To protect the infrastructure, a documented security code review report is required for any externally facing or demilitarized zone (DMZ)-hosted technology solution containing custom programming or scripting, regardless of the designation of sensitivity. A security code review report is also required for technology solutions designated as sensitive that contain active content code or CGI scripts.
The security code review will be based on the Postal Service Information Security Code Review Standards or an acceptable equivalent (This security code review will not be required if a CISO approved independent security code review is conducted).
10. Conduct Vulnerability Scans
Vulnerability scans are recommended for all applications and are required for the following applications:
a. Annually for externally facing applications.
b. Quarterly for PCI applications (i.e., applications utilizing credit card transactions).
Scanning procedures must ensure adequate scan coverage and the updating of the list of vulnerabilities to be scanned for.
11. Test Security Control
This step will focus on testing the security controls and processes acquired, built, and integrated during the Build Phase to determine their effectiveness. All test results are documented on the scripted materials, identified as problems on the problem log, or identified as new requirements on the enhancement log. All test results should be retained in accordance with Postal Service data retention requirements. An applications security test and evaluation plan must be developed for applications designated as sensitive. The Security Test and Evaluation (ST&E) Plan defines the security testing to be conducted to determine the extent to which the application meets the security requirements for its mission and operational environment. Since the ST&E Plan is part of an overall system test plan, highlight or flag the security section for ease of review. Sensitive test data should be protected throughout the entire testing cycle.
This process will include the following steps:
a. Conducting security testing and documenting the results.
b. Conducting vulnerability scans.
c. Addressing any outstanding issues, if applicable.
d. Include both the technical and non-technical security controls.
e. Include controls associated with hardware, operating system, networking and telecommunications, physical security, personnel security, and computer operations, and manual processes.
f. Define the security functionality (security control feature) to be tested for each security control implemented to satisfy the security requirement.
g. Describe the actual testing to be performed for each control. For each control, include applicable test scripts, scenarios, performance thresholds, and an indication of what will constitute passing or failing.
h. The core team develops an application ST&E results report and submits the findings to the executive sponsor. If the ST&E report is a part of an overall test report, highlight or flag the section addressing information security testing for ease of review.
5 PROCESS DIAGRAMS
6 PROCESS INPUTS/OUTPUTS
Inputs
1. Technology Solution Requirements / Approved - All functional, business, and technical requirements for designing the technology solution.
2. Updated Traceability Test Matrix - Updated Traceability Test Matrix from Build documentation.
3. Technology Solution Design / Approved - Approved Technology Solution Design document that encompasses all the components that make up the structure and overall functionality of the technology solution. This technology solution includes all functional engineering elements and relationships between them.
4. Project Plan Final Baseline The project plan encompassing the full lifecycle of the Technology Solution.
5. Draft SIT Strategy - Draft strategy documented in Analysis & Design.
6. TS Environment Build - Technology solution environment ready for test phase (e.g. SIT/CAT/PROD).
7. Build Plan - Plan to accomplish the Build phase.
Outputs
1. SIT Plan - Final SIT plan with completed milestones.
2. SIT Readiness Plan A project plan that addresses SIT activities such as room and log setup.
3. Updated Traceability Test Matrix - Updated Traceability Test Matrix with SIT test procedures and test results.
4. Final SIT Strategy - Final SIT strategy document.
5. SIT Documented Approval Documented means of indicating that SIT was successfully completed.
6. Other Documentation:
a. SIT Test Results SIT test results
b. Post-Implementation Review Problems detected and enhancement requested from SIT and lessons learned from this testing
7. Security Code Review Report Review of code to uncover security risks.
8. Security Test Results Documentation of security test results from the Security Test and Evaluation.
7 PROCESS RELATIONSHIPS
8 ROLES AND RESPONSIBILITIES
Technology Solution Coordinator (TSC) Responsible for coordination and execution of the SIT and its deliverables within the approved cost and schedule. This role may be the responsibility of any TSP but is typically performed by IBSSC.
Technology Solution Project Manager (TSPM) Responsible for managing the overall project. This role may be the responsibility of any TSP but is typically performed by the Portfolio.
Technology Service Providers (TSP) All Stakeholders that are responsible for testing their system components of the technology solution.
